# `build_synthesis_prompt()`

你现在处于 project distill pipeline 的 reduce / synthesis 阶段。

你会收到一份 synthesis packet。它由多个 chunk reports 拼成。  
你的任务不是拼接这些 reports，而是做跨 chunk 综合、去重、筛选和定稿。

## 先做 chunk coverage planning（内部先做，不一定展示）

在正式写任意最终文档前，你必须先在内部完成这一步：

1. 识别每个 chunk 的主题。
2. 提取每个 chunk 的高价值信号，尤其优先看 `priority_for_synthesis: high` 的信号。
3. 区分哪些信号属于：
   - 项目主干
   - 阶段性工作
   - 一次性问题
   - 可迁移经验
   - 协作建议
4. 检查每个 chunk 是否至少有高价值信号被认真考虑。
5. 明确哪些信号应进入当前文档，哪些应降权或丢弃。

不要让最新 chunk、最长 chunk、最具体 chunk 自动获得最高权重。
如果某个 chunk 的高价值信号没有进入当前文档，应该是因为它不适合当前文档，而不是被忽略。

一般来说：
- 更早出现、且定义项目定位/边界/主实现路线的信号，默认比后期局部工程细节更接近“项目主干”；
- 后期更具体的发布、展示、局部修复、环境问题，通常应写成“近期重点”或“阶段性工作”，不应替代整个项目定义。

你必须一次性输出一个 **3 文档 bundle**，其中包含：

1. `project_context.md`
2. `collaboration_guide.md`
3. `patterns.md`

## 文档 1：`project_context.md`

你现在要生成 `project_context.md`。

### 文档定位

`project_context.md` 是给未来接手本项目的 agent / 开发者看的项目上下文。

它回答：

- 这个项目是什么；
- 核心目标与非目标是什么；
- 关键边界是什么；
- 关键决策是什么；
- 哪些文件、模块、配置重要；
- 怎么验证；
- 有哪些环境约束和已知坑。

### 只应该写

- 项目目标与非目标；
- 架构/产品边界；
- 关键设计决策；
- 重要文件、模块、配置；
- 验证方式；
- 环境约束；
- 已知坑；
- 对外发布或文档的项目级要求；
- 未来 agent 接手时必须知道的项目事实。

### 不要写

- 用户性格判断；
- 用户下次应该怎么提需求；
- agent 如何和用户沟通的长篇建议；
- 泛用 workflow 总结；
- 没有证据的长期偏好；
- 当前项目无关的通用经验。

### 多 chunk 综合规则

你必须先做跨 chunk 综合：

1. 去重  
同一事实出现在多个 chunk，只写一次，并合并证据。

2. 分级  
多个 chunk 支撑的事实，可以高置信写入。  
单 chunk 支撑但证据强的事实，可以写入，但语气要克制。  
单 chunk 支撑且像偶然现象的内容，不写或放到低置信 notes。

3. 阶段区分  
区分：
- 最终项目事实；
- 中间讨论过但后来放弃的方案；
- 一次性环境问题；
- 仍未确定的方向。

不要把早期 chunk 的中间方案误写成最终项目事实。

4. 冲突处理  
如果不同 chunk 对同一问题有冲突，要显式说明当前更可信的版本，或标成未定。不要悄悄选择一边。

5. 避免 report 拼贴  
最终文档必须是综合后的项目上下文，不要按 chunk 顺序复述。

6. 先写稳定主干，再写近期上下文  
不要用某个 chunk 的近期任务替代整个项目定位。先交代项目稳定主干，再补近期阶段性重点。

### 建议结构

# Project Context

## Overview
## Goals and non-goals
## Key decisions
## Important files and modules
## Validation approach
## Environment constraints
## Known gotchas
## Notes for future agents

### 写作要求

- 中文输出。
- 在 bundle 中，这一节只输出 `project_context.md` 的正文。
- 必须先写稳定项目定位、安全边界、核心设计，再写近期阶段性重点。
- 不要让某个 chunk 的近期工作（例如发布、展示、局部工程问题）替代整个项目主干。
- 如果较早 chunk 包含项目定义、主边界、核心设计，而较晚 chunk 更像近期收尾/包装/排障，前者默认应先写、后者后写。
- 每条都要像“项目事实/项目规则”，不要像复盘感想。
- 能具体就具体，例如文件名、配置名、命令、模式名、平台限制。
- 重要结论尽量绑定 chunk id / report 证据。
- 证据弱的内容不要写，或放到 Notes 中标明低置信。

## 文档 2：`collaboration_guide.md`

你现在要生成 `collaboration_guide.md`。

### 文档定位

`collaboration_guide.md` 是给用户和未来 agent 共同看的协作指南。

它不是固定格式的流程表，也不是通用沟通建议。
它的任务是：**从 chunk reports 里自然发现这个项目中最值得保留的协作方式，并把它写成下一次协作能直接用的指南。**

它回答：

* 这个项目里，用户和 agent 怎么配合更顺；
* 哪些协作方式明显减少了返工；
* 哪些地方容易误解、跑偏或重复确认；
* 用户下次可以怎么说；
* agent 下次应该主动确认什么；
* 哪些建议可以由 agent 代劳提醒，而不是全靠用户记住。

### 只应该写

* 从真实对话中发现的高价值协作方式；
* 用户下次可直接复用的话术；
* agent 下次应主动提出的问题或提醒；
* 某类任务中容易造成返工的点；
* 能让下一次协作更快收敛的具体动作；
* 有证据支撑的项目内协作习惯或工作方式。

### 不要写

* 项目架构细节长篇总结；
* 把用户描述成固定人格；
* 泛泛建议，如“表达清楚一点”“多沟通”；
* 没有证据的通用沟通技巧；
* 为了填栏目而硬凑内容；
* 只给用户增加记忆负担、但 agent 无法代劳的建议。

### 多 chunk 综合规则

你必须先做跨 chunk 综合：

1. **先发现，再组织**

不要预设固定栏目。
先判断 chunk reports 中最强的协作信号是什么，再根据这些信号自然组织文档结构。

section 标题应该由内容决定，而不是套模板。

2. **去重**

多个 chunk 中反复出现的协作摩擦，只保留一条综合建议。
不要把同一个建议拆散到多个小标题下重复写。

3. **分级**

多个 chunk 都支持的协作方式，可以高置信写入。
单 chunk 支撑的建议，必须写清适用场景，不能写成长期偏好。

4. **用户建议与 agent 代劳并列**

重要建议尽量同时写：

* 用户可直接说的话；
* agent 应主动问的话；
* 为什么这样能减少返工。

不要只告诉用户“下次记得说清楚”。
能由 agent 主动确认的，就写成 agent 的责任。

5. **避免人格化**

不要写：

* “用户很敏感”
* “用户总是……”
* “用户偏好……”

更好的写法是：

* “在设计/边界类任务中，先确认目标和非目标可以减少返工。”
* “在发布展示类任务中，先区分 README、包元数据和预览图可以避免误判。”

6. **覆盖不同任务阶段**

如果 chunk reports 中同时存在设计/边界确认、实现/验证、发布/展示等不同阶段的强信号，这份文档不能只覆盖其中一个阶段。

不要因为某个阶段更具体、更新，就让它主导全文。

### 推荐组织方式

不要固定使用下面这些标题。
你可以根据 chunk reports 自己命名 3-6 个 section。

每个 section 最好围绕一个真实协作场景或高价值协作原则，例如：

* 先判断路线，再动手
* 先钉住目标、非目标和边界
* 安全/沙箱类任务如何分工验证
* 发布/展示类任务如何避免误判
* 用户可以直接给出的高价值指令
* agent 应主动承担的确认工作

每个 section 建议包含：

* 适用场景：什么时候用；
* 用户可说：用户下次可以直接复制的话；
* Agent 应问：agent 应主动确认什么；
* 为什么：它解决了什么返工或误解；
* 证据：绑定 chunk id / report 证据。

如果某个 section 不适合这个格式，可以调整，但必须保持具体、可执行、有证据。

### 写作要求

* 中文输出。
* 在 bundle 中，这一节只输出 `collaboration_guide.md` 的正文。
* 结构要自然，不要为了整齐而牺牲内容发现。
* 每条建议尽量能直接进入下一次对话。
* 每条建议要有适用场景。
* 重要结论尽量绑定 chunk id / report 证据。
* 不要把单 chunk 现象说成长期用户偏好。
* 宁可少写，也不要写泛泛沟通建议。

### 示例写法

```markdown
## 先判断路线，再动手

适用场景：设计审查、发布展示、沙箱边界、网络能力等容易走错方向的任务。

用户可说：
- 先给我路线判断和风险点，不要急着改。
- 先判断这是 README、包元数据，还是 pi.dev 预览图问题。

Agent 应问：
- 这是先讨论方案，还是可以直接实现？
- 这次要改的是文案、元数据，还是运行行为？

为什么：
- 这类任务如果一开始误判对象，后面容易产生无效修改和返工。

证据：
- chunk_001 / chunk_002 中多次出现先判断路线再执行的强信号。
```


## 文档 3：`patterns.md`

你现在要生成 `patterns.md`。

### 文档定位

`patterns.md` 是从本项目中提炼出的可迁移经验。

它回答：

- 哪些做法不只对本项目有用，也可能在未来类似项目中复用；
- 这些做法适用于什么场景；
- 具体怎么做；
- 为什么有用；
- 不适用于什么情况。

### 只应该写

- 可迁移的开发模式；
- 安全边界类功能的经验；
- 测试/验证模式；
- 产品化/发布/文档模式；
- 第三方依赖处理经验；
- 这些模式的适用条件和不适用条件。

### 不要写

- 第二份项目总结；
- 用户协作建议；
- 太抽象的口号；
- 没有适用条件的万能规则；
- 单次偶然问题的过度泛化；
- 没有未来复用价值的历史细节。

### 多 chunk 综合规则

你必须先做跨 chunk 综合：

1. 合并重复 pattern  
同一类模式在多个 chunk 中出现，要合并成一个 pattern，并合并证据。

2. 分清项目事实和可迁移经验  
“本项目用了 X”不是 pattern。  
只有当它能抽象成“类似场景下可以怎么做”时，才写入 patterns。

3. 必须写适用条件  
每个 pattern 都必须写：
- 适用场景；
- 具体做法；
- 为什么有用；
- 证据；
- 注意边界。

4. 防止万能化  
如果一句话放到任何项目都成立，说明它太抽象，需要改写或删除。

5. 保留 caveats  
对于 fork/vendor、fail-closed、真实测试等强建议，必须写清不适用情况。

6. 保持类型覆盖  
patterns 应覆盖不同类型的可迁移经验；不要只提取最新、最具体、最容易展开的工程问题。

### 建议结构

# Patterns

## Reusable patterns

### Pattern：一句话标题
- 适用场景：
- 具体做法：
- 为什么有用：
- 证据：
- 注意边界：

## Caveats

### 写作要求

- 中文输出。
- 在 bundle 中，这一节只输出 `patterns.md` 的正文。
- 应同时考虑安全边界、真实验证、runtime 副作用处理、发布展示、环境/配置修复等不同类型的可迁移经验。
- 不要只提取后期 chunk 中最具体的几个工程问题。
- 如果较早 chunk 包含更底层、更可迁移的 pattern，而较晚 chunk 只是更具体的近期问题，前者不应被后者压掉。
- 优先保留既具体又有跨场景复用价值的 pattern。
- 重要结论尽量绑定 chunk id / report 证据。
- 优先保留能影响未来行动的经验。
- 宁可少写，也不要写万能鸡汤。

## 全局规则

- 只基于 synthesis packet 中的 chunk reports 和元数据。
- 不要读取其他文件。
- 不要编造未出现的事实。
- 不要把单个 chunk 的现象说成长期规律。
- 不要把中间方案误写成最终事实。
- 重要结论尽量绑定 chunk id / report 证据。
- 低相关 chunk 只能低权重参考。
- 三份文档要分工清楚，避免互相重复。
- 要区分项目主干与阶段性工作；不能把近期工作写成整个项目。
- 如果某个 chunk 明显包含高价值信号，最终三份文档整体上不应几乎完全没有体现。
- 如果证据不足，宁可不写。
- 只输出 bundle，不要解释生成过程。
